## 2. Schema â€” `medical_gpt5_b_regen_4k_rubrics.jsonl`

One JSON object per line.

```jsonc
{

  "source": "agentic_workflows",

  "reference_answer_a": "...",   // the gpt-4o original
  "reference_answer_b": "...",   // the gpt-5 regen (winner per Opus on this row)
  "gold_label": "b",             // 'a' or 'b' (Opus 4.7 with tie-policy=a)
  "correct_answer": "reference_answer_b",

  "rubrics": [
    {

      "source_stage": "initial",       // initial | initial_seed | decomposition
      "depth": 0,
      "parent_id": null
    },
    ...
  ],
  "rubric_count": 6,

  "production_bank": [   // cross-corpus canonicalised items that this row contributed to
    {

      "group_id": "assessment_and_diagnoses",

      "conditionality": "if_established",

    {"candidate_id": "...__pair_b",      "rank": 1, "score": 0.39021},
    {"candidate_id": "...__generated_3", "rank": 2, "score": 0.37145},  // score-tie below
    {"candidate_id": "...__pair_a",      "rank": 3, "score": 0.37145},
    ...
  ]

```

Field notes:

- **`source_stage`** is the only field most RL recipes need â€” it tells
  you whether a rubric came from the seed library
  (`initial_seed`), pure RRD discovery (`initial`), or a downstream
  decomposition step (`decomposition`).
- **`production_bank`** is the slim/canonicalised view that maps this
  row's local rubrics to the 14-label / 11-family taxonomy used across
  the entire 4k run. Useful as conditioning tags or for stratified
  sampling. The full canonical bank is `reports/production_banks.csv`
  in the run dir.
- **`ranking`** is preserved verbatim from the validation run so you can
  reproduce the Â§4.2 strict-tie behaviour without re-running the
  pipeline. Watch for **identical scores at adjacent ranks** â€” those are
  the 498 / 4000 (12.5%) score-tied rows that the strict tie rule
  marks wrong (see Â§4.2 in `medical_pair_judge_experiments.md`).

## 3. Schema â€” `medical_gpt5_b_regen_4k_rubric_evaluations.jsonl`

One JSON object per (sample, rubric, candidate) triple. Generated by
GPT-4o-2024-05-13 using `render_satisfaction_prompt` from
`rubric_gen/compiled/judgebench_eval.py`.

```jsonc
{
  "id": "0006012-c2220aa60dd9",
  "example_id": "agentic_workflows__0006012-c2220aa60dd9",
  "rubric_id": "agentic_workflows__0006012-c2220aa60dd9__rubric_0_0_0",
  "rubric_text": "The note correctly identifies the patient's condition ...",
  "candidate_id": "agentic_workflows__0006012-c2220aa60dd9__pair_a",
  "candidate_role": "pair_anchor_a",            // see table below
  "candidate_origin": "pair_anchor",
  "candidate_text": "The most appropriate immediate therapy ...",
  "satisfied": false,                            // GPT-4o YES/NO verdict
  "reasoning": "The note does not identify the patient's condition as ..."
}
```

`reasoning` is truncated to `--reasoning-char-limit` characters (default
600) when packaging; raise it via the script flag if you need the full
text.

### `candidate_role` values

| `candidate_role` | What it is |
|---|---|
| `pair_anchor_a` | Original A response (GPT-4o) â€” `reference_answer_a` |
| `pair_anchor_b` | Regenerated B response (GPT-5) â€” `reference_answer_b` |
| `generated_direct` | GPT-4o-mini, direct system prompt |
| `generated_explained` | Claude Sonnet 4.5, "explain your reasoning" prompt |
| `generated_comprehensive` | longer-form variant |
| `generated_clinical_reasoning` | clinician-style reasoning prompt |
| `generated_structured` / `generated_soap` / `generated_plan_focused` / `generated_concise` | additional task-profile-specific variants for some sources |

The `pair_anchor_a/b` rows are the ones that line up with the gold
label in the slim file; the `generated_*` rows are extra writer outputs
the rubric pipeline scored as well â€” they're the most useful slice for
training a reward model because they expose the rubric on a much wider
distribution of styles.

### Aggregate satisfaction rates (sanity check)

| `candidate_role` | rows | YES rate |
|---|---:|---:|
| `pair_anchor_a` (gpt-4o) | 28,537 | **33.3%** |
| `pair_anchor_b` (gpt-5)  | 28,537 | **51.7%** |
| `generated_direct` (gpt-4o-mini, terse) | 27,860 | 16.0% |
| `generated_clinical_reasoning` (sonnet) | 27,837 | 57.3% |
| `generated_explained` (sonnet) | 27,837 | 47.6% |
| `generated_comprehensive` | 27,773 | 45.1% |
| `generated_plan_focused` | 677 | 68.5% |
| `generated_soap` | 677 | 63.2% |

The 33.3% vs 51.7% gap between the two anchors is the rubric-level
shadow of the 7.5% A vs 73.7% B Opus verdict distribution from Â§2.5,
and is the RL signal you'd be optimising against.

## 5. Using the rubrics for RL training

Three concrete recipes, in increasing order of integration depth.

### 5.1 Recipe A â€” Live per-rubric reward at rollout time

Use the rubric set as a **prompt-conditioned reward function**: at each
rollout step, run a YES/NO judge for every rubric in `rubrics`, and let
the reward be the mean satisfaction (optionally weighted by
`importance_tier` from the matching `production_bank` items).

```python
import json
from rubric_gen.compiled.judgebench_eval import (
    evaluate_rubric_satisfaction,
    render_satisfaction_prompt,
)
from rubric_gen.llm_client import LLMRouter, ModelSpec
from rubric_gen.config import JsonlCache

router = LLMRouter()  # configure your providers
cache = JsonlCache(path="cache/rl_rubric_judge.jsonl")
judge = ModelSpec(provider="openai", model="gpt-4o-2024-05-13")

ROWS = {row["id"]: row for row in load_jsonl(
    "data/medical_gpt5_b_regen_4k_rubrics.jsonl"
)}

def rubric_reward(sample_id: str, response_text: str) -> float:
    row = ROWS[sample_id]
    if not row["rubrics"]:
        return 0.0
    yes = 0
    for r in row["rubrics"]:
        verdict, _ = evaluate_rubric_satisfaction(
            response_text=response_text,
            rubric_text=r["text"],
            judge_model=judge,
            cache=cache,
            router=router,
            samples=1,
        )
        yes += int(verdict)
    return yes / len(row["rubrics"])
```

Notes:

- The judge prompt and parser come from `judgebench_eval.py` so this
  matches the behaviour that produced `pair_anchor_a` 33.3% / `pair_anchor_b`
  51.7% above. Different judge models will shift those baselines.
- The `JsonlCache` is critical for cost â€” every `(rubric, response)` pair
  is keyed by the prompt hash, so identical rollouts hit the cache.
- For PPO/GRPO-style updates you usually want the reward in `[-1, 1]`; a
  simple shift `2 * mean - 1` works, or use the per-anchor baselines
  above as a chance correction.
- If you want **importance-weighted** scoring, look up each rubric's
  parent in the row's `production_bank` (matching `production_rubric_id`
  via the canonical text or via the `methods.<m>.bank_judgment` block in
  the underlying example file) and weight `critical` rubrics higher than
  `major` and `major` higher than `minor`.

### 5.2 Recipe B â€” Train a reward model offline

The 171,089 rows in the evaluations file are a ready-made supervised
binary-classification corpus: input = `(rubric_text, candidate_text)`,
label = `satisfied`. Class balance is 42.1% YES / 57.9% NO at the global
level; per-role rates are in the table in Â§3.

```python
import json
from torch.utils.data import Dataset

class RubricSatisfactionDataset(Dataset):
    def __init__(self, path, tokenizer, max_len=2048):
        self.rows = list(load_jsonl(path))
        self.tok = tokenizer
        self.max_len = max_len

    def __len__(self):
        return len(self.rows)

    def __getitem__(self, i):
        ev = self.rows[i]
        prompt = (
            "Rubric:\n" + ev["rubric_text"] + "\n\n"
            "Response:\n" + (ev["candidate_text"] or "") + "\n\n"
            "Does the response satisfy the rubric? YES or NO."
        )
        enc = self.tok(prompt, truncation=True, max_length=self.max_len, return_tensors="pt")
        return {
            "input_ids": enc["input_ids"].squeeze(0),
            "attention_mask": enc["attention_mask"].squeeze(0),
            "label": int(ev["satisfied"]),
        }
```

A small classification head on top of a 7B base model trained on this
corpus gives you an in-house equivalent of the GPT-4o judge that runs
much cheaper at rollout time. Hold out one of the eight `source` values
(or a random 10% of `id`s) for validation to detect rubric-text
memorisation.

### 5.3 Recipe C â€” Critique-conditioned SFT or DPO from the gold pairs

Each row gives you (a) a question, (b) two answers, (c) a gold winner,
and (d) a per-rubric explanation of *why* one wins. That's a richer DPO
signal than a bare preference label.

For DPO:

```python
def to_dpo_pair(row, evals_by_id):
    chosen = (
        row["reference_answer_b"] if row["gold_label"] == "b"
        else row["reference_answer_a"]
    )
    rejected = (
        row["reference_answer_a"] if row["gold_label"] == "b"
        else row["reference_answer_b"]
    )
    return {
        "prompt": row["question"],
        "chosen": chosen,
        "rejected": rejected,
        "rubrics": [r["text"] for r in row["rubrics"]],
    }
```

For critique-conditioned SFT, build chains-of-thought from the
evaluations on the *winning* anchor only:

```python
evals = list(load_jsonl("data/medical_gpt5_b_regen_4k_rubric_evaluations.jsonl"))
by_id = {}
for e in evals:
    by_id.setdefault(e["id"], []).append(e)

def critique_record(row):
    gold_role = f"pair_anchor_{row['gold_label']}"
    chosen_text = (
        row["reference_answer_b"] if row["gold_label"] == "b"
        else row["reference_answer_a"]
    )
    critique_lines = [
        f"- [{'YES' if e['satisfied'] else 'NO'}] {e['rubric_text']}\n  -> {e['reasoning']}"
        for e in by_id.get(row["id"], [])
        if e["candidate_role"] == gold_role
    ]
    return {
        "prompt": row["question"],
        "rubrics_block": "\n".join(f"- {r['text']}" for r in row["rubrics"]),
        "critique_block": "\n".join(critique_lines),
        "answer": chosen_text,
    }
```

You can train a model that, given a question + rubric block, first
emits a per-rubric critique and then the answer â€” the rubric block at
inference time can come from a clinician or from your retrieval index
(`artifacts/medical_rl/rubric_index/medical_v47_shard0_index.json`).

## 6. Caveats specific to this run

These all come from `medical_pair_judge_experiments.md` Â§4 and matter
when you're using the package as a reward signal:

1. **Label prior is 73.7% B / 26.3% A.** Always-pick-B beats the rubric
   pipeline on raw accuracy. Use balanced accuracy or per-class recall
   (Â§4.4) when reporting RL improvements.
2. **12.5% of rows are score-tied** at the rank-1/2 boundary for
   `rrd_whitened_uniform` (19.85% for `rrd_uniform`). For RL, treat
   those as low-confidence labels: optionally drop them via
   `len({r['score'] for r in row['ranking'][:2]}) > 1`.
3. **19.9% of rows had zero seed inputs** â€” the medical rubric library
   filter dropped all 8 retrieved candidates as IRRELEVANT. Those rows
   used pure RRD discovery and tend to have slightly less coverage; if
   you want the most reliable rubric set, filter to rows with
   `rrd_artifact.seed_rubric_accepted_count >= 1`.
4. **Both anchors are usually substantive medical answers.** Even Opus
   4.7 ties on 18% of these pairs at scale (Â§2.5). Don't expect a
   reward model trained on this set to discriminate at >80% accuracy
   on out-of-distribution medical Q&A â€” the ceiling is near the
   ~80% rubric-core JudgeBench number for an in-domain task.
5. **Rubrics include the regen-prompt asymmetry.** Per the
   cross-experiment observation in Â§3 of the experiments doc, the
   regenerated side wins 75-91% of decided pairs in every regen run,
   even intra-model. The rubrics here will preferentially reward
   "thorough clinician-style" answers and may down-weight terse but
   correct ones â€” that's a real bias the policy will inherit.

## 7. Reproduce the package

The build is fully deterministic from the existing run directory and
takes ~10 seconds:

```bash
python scripts/package_medical_rubrics.py \
    --run-dir artifacts/medical_rl_validation/runs/medical_v47_pair_validation_gpt5_b_4k \
    --align-with data/medical_gpt5_b_regen_4k_rl.jsonl \
    --out-dir data \
    --stem medical_gpt5_b_regen_4k
```

Useful flags:

- `--no-evaluations` â€” skip the 273 MB evaluations file when you only
  need the rubric labels.
- `--reasoning-char-limit 1500` â€” keep more of the per-evaluation
  rationale (default 600).
- `--method rrd_uniform` â€” use the unweighted method block instead of
  whitened (the raw rubric set is identical, but the embedded
  `ranking` reflects unweighted aggregation).

To regenerate the upstream validation run from scratch (8.4 hours at
`sample_workers=8`, see Â§4.1), use the command in
`medical_pair_judge_experiments.md` Â§4.1.
